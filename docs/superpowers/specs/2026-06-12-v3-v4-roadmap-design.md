# v3 / v4 로드맵 설계 — Share Core & Community

- 작성일: 2026-06-12
- 상태: 사용자 승인된 목표 설계 (구현 계획은 별도 문서)
- 선행: v1 MVP (M0–M6), v2 Visual Major (V2-M0–M6), SoC Custom Studio — 모두 `main`에 머지 완료

## 배경과 결정 요약

v2까지는 local-first 단일 사용자 창작 도구로서의 비주얼 품질에 집중했다. v3/v4의 핵심 가치는
**공유·커뮤니티**로 결정했다. 사용자와의 브레인스토밍에서 확정된 결정:

| 결정 항목 | 결정 |
|---|---|
| v3 방향 | 공유·커뮤니티 (표현력 확장·모바일·3D·AI보다 우선) |
| v3 범위 | 공유 코어만 — 백엔드 + 계정 + 퍼블리시 + 공개 갤러리 + 공유 링크 |
| v4 방향 | 커뮤니티 완성 — 반응/랭킹/콘테스트/리믹스 계보 |
| 백엔드 형태 | Node + TypeScript + SQLite 자체 백엔드 (BaaS/서버리스 아님) |
| 공개 배포 | v3은 **배포 가능 상태**까지. 실제 공개 오픈은 별도 결정 (v4 시작 시 게이트) |
| 공유 모델 | **Publish 스냅샷 모델** — 양방향 동기화 없음 |

## v3 — "Share Core" (공유 코어)

### 한 줄 목표

편집은 지금처럼 100% 로컬(IndexedDB)로 유지하면서, 완성한 칩을 계정 기반으로 퍼블리시해
공개 갤러리와 공유 링크로 보여줄 수 있는 백엔드를 구축한다. v3 완료 기준은 실제 공개 오픈이
아니라 **배포 가능 상태**다.

### 아키텍처 핵심 결정

- **Publish 스냅샷 모델.** 서버는 편집 상태를 알지 못한다. 퍼블리시 시점에 클라이언트가
  (1) 프로젝트 JSON 스냅샷과 (2) 클라이언트에서 렌더한 PNG(die-only + poster)를 업로드한다.
  서버에서 Konva 재렌더를 하지 않으므로 서버사이드 canvas 의존이 생기지 않는다.
  동기화/충돌 해결이 없고 local-first 계약(오프라인 편집 포함)이 그대로 유지된다.
- **워크스페이스 구조.** npm workspaces로 `server/`를 추가한다. 기존 `src/domain/`의
  프로젝트 JSON schema·검증·마이그레이션 로직은 순수 모듈이므로 서버가 재사용한다.
  domain 순수성 규칙(React/Konva/Zustand/IndexedDB/browser import 금지)은 서버 공유의
  전제 조건으로 계속 유지한다.
- **단일 JSON schema 계약 유지.** published 스냅샷도 로컬 프로젝트와 같은 schema와
  `schemaVersion`을 쓴다. 서버는 schema 검증·저장만 담당하고 내용을 변형하지 않는다.
  스냅샷의 schema가 구버전이면 클라이언트(가져오기 시점)에서 기존 마이그레이션을 적용한다.
- **인증.** 서명된 세션 쿠키 + argon2id 비밀번호 해시. JWT는 사용하지 않는다 (단일 서버,
  세션 무효화 단순성).
- **이미지 저장.** PNG는 서버 디스크에 파일로 저장하고 DB에는 메타데이터만 둔다.
- **DB.** SQLite (마이그레이션 도구 포함). 서버 스택 세부(프레임워크, ORM/쿼리 계층)는
  구현 계획 단계에서 확정한다.

### v3 완료 기준

- 로컬 환경에서 두 개의 계정으로 **가입 → 퍼블리시 → 갤러리 열람 → 공유 링크 열람 →
  리믹스 가져오기 → 로컬 편집** 플로우가 끊김 없이 동작한다.
- 서버를 끈 상태에서도 기존 로컬 편집/저장/export 흐름에 회귀가 없다 (local-first 유지).
- 갤러리/뷰어 페이지가 v2 페이지 테마 시스템과 시각 언어를 공유한다 (비주얼 품질 게이트는
  v3에도 적용 — 공유 페이지가 어설프면 공유 가치가 없다).
- `npm test`와 `npm run build`가 클라이언트·서버 모두 green이고, 배포 문서/스크립트가 존재한다.
- 실제 공개 오픈은 하지 않는다 (기존 비공개 계약 유지).

### v3 에러 처리 원칙

- 퍼블리시 실패(네트워크/서버 다운)는 로컬 데이터에 어떤 영향도 주지 않는다.
- 서버는 모든 입력(JSON schema, 이미지 크기/형식, 필드 길이)을 검증하고 잘못된 입력을
  명확한 에러로 거절한다.
- 클라이언트는 서버 부재를 정상 상태로 취급한다 — 공유 기능만 비활성화되고 나머지는 동작.

### v3 테스트 원칙

- 기존 컨벤션(TDD, Vitest, Konva는 브라우저 검증) 유지.
- 서버는 API 레벨 통합 테스트(인메모리/임시 SQLite) + 순수 로직 단위 테스트.
- domain schema 검증은 클라이언트·서버가 같은 테스트를 공유한다.

## v4 — "Community" (커뮤니티 완성)

### 한 줄 목표

v3 공유 인프라 위에 반응·랭킹·콘테스트·리믹스 계보를 쌓아 갤러리를 살아있는 커뮤니티로
만든다. 실제 공개 오픈 여부는 v4 시작 시 별도 게이트로 결정한다.

### 마일스톤

| 마일스톤 | 내용 |
|---|---|
| **V4-M0** | 운영 준비: 관리자 롤, 신고/숨김/삭제 등 모더레이션 기본기 + 공개 오픈 결정 게이트 |
| **V4-M1** | 반응: 좋아요 + 댓글 + 신고 |
| **V4-M2** | 랭킹/트렌딩: 주간/전체 정렬 (단순 점수 산식, 과한 알고리즘 금지) |
| **V4-M3** | 콘테스트: 주제 공모 → 기간 출품 → 투표 → 결과 발표 (결과 화면은 키노트 포스터 톤) |
| **V4-M4** | 리믹스 계보: published 칩의 remix lineage 트리 시각화 — 칩 미학과 어울리는 v4 시그니처 비주얼 기능 |

### v4 범위 결정

- **일반 게시판은 만들지 않는다.** 칩 상세의 댓글 + 콘테스트 공지로 흡수한다.
  제품 정체성이 "칩이 주인공"이므로 칩과 분리된 텍스트 게시판은 YAGNI.
- v4 세부 설계(점수 산식, 콘테스트 규칙, lineage 데이터 모델)는 v3 완료 후
  별도 브레인스토밍/스펙으로 확정한다. 이 문서의 v4 부분은 방향 합의이지 구현 스펙이 아니다.
